# MSCL 지식 위키 — 운영 규칙 (Schema)

## 위키 소개

**운영자**: 서포트팀 이사 (유아용품 수입·제조·판매 전문 일본 법인)
**목적**: 업무와 학습에서 쌓이는 지식을 축적하고, 필요할 때 즉시 꺼내 쓰는 나만의 나무위키
**맥락 파일**: 추가 예정 (`raw/context-mission.md` 등록 시 이 섹션 업데이트)

---

## 폴더 구조

```
MSCL/
├── CLAUDE.md          # 이 파일 — 위키 운영 규칙
├── raw/               # 원본 자료 (사용자가 넣음, Claude는 읽기만)
└── wiki/              # 정리된 지식 (Claude가 작성·관리)
    ├── index.md       # 전체 목차
    └── log.md         # 작업 이력
```

**플랫 구조 원칙**: `raw/`와 `wiki/` 안에 하위 폴더 없이 파일만 보관. 분류는 `wiki/index.md`에서 카테고리별로 관리.

---

## 파일 명명 규칙

| 유형 | 패턴 | 예시 |
|------|------|------|
| raw 원본 | 날짜-키워드.확장자 | `2026-05-01-반품정책-기사.md` |
| wiki 페이지 | 키워드.md (한글 허용) | `반품정책.md`, `재고관리-노하우.md` |

---

## 위키 페이지 형식

모든 위키 페이지는 아래 구조를 따른다:

```markdown
# 페이지 제목

> [!tip] 서포트팀 이사 관점 핵심 takeaway
> 이 내용에서 **내가 지금 당장 써먹을 수 있는 것**, 또는 우리 사업(유아용품 일본 법인)에
> 직접 영향을 주는 포인트를 2–3문장으로 요약. 추상적 설명 금지, 구체적 행동/판단 기준으로.

---

## 개요
(내용 요약)

## 핵심 내용
(구체적 내용)

## 다운로드
> 양식 파일이 raw/에 있을 경우에만 포함. 없으면 이 섹션 생략.
- [[raw/파일명.docx|⬇ 파일명 다운로드]]

## 관련 페이지
- [[관련페이지1]]
- [[관련페이지2]]

## 출처
- raw/원본파일명.md (날짜)
```

---

## 운영 방법 3가지

### 1. 자료 넣기 (Ingest)

**사용자가 할 일**: `raw/` 폴더에 파일을 넣고 Claude에게 알린다.
> "raw/파일명 읽고 위키로 만들어줘"

**Claude가 할 일**:
1. `raw/파일명` 읽기
2. 연관된 기존 wiki 페이지 파악 (index.md 참조)
3. 새 페이지 생성 또는 기존 페이지 업데이트 (보통 1–5개)
4. 모든 페이지에 [[위키링크]] 교차참조 추가
5. `wiki/index.md` 목차 업데이트
6. `wiki/log.md`에 작업 이력 추가

**규칙**:
- 원본 `raw/` 파일 내용은 수정하지 않는다. 단, ingest 완료 후 **파일 맨 위에 처리 완료 마커 한 줄을 추가**한다:
  ```
  > ✅ wiki 반영 완료 → [[wiki페이지명]]
  ```
  - YAML frontmatter가 있는 파일은 닫는 `---` 바로 다음 줄에 추가
  - frontmatter가 없는 파일은 첫 줄에 추가
  - 이 마커가 Obsidian 그래프에서 raw↔wiki 연결선을 만든다
- 페이지 첫 블록은 반드시 `> [!tip] 서포트팀 이사 관점 핵심 takeaway` callout
- 교차참조는 적극적으로: 같은 개념이 다른 페이지에 있으면 반드시 [[링크]]

**내용 우선 통합 원칙** ⚠️ 항상 적용:
- **관련 기존 wiki 페이지가 있으면 무조건 통합** — 새 페이지를 만들지 않고 해당 페이지에 내용을 추가한다
  - raw 파일을 저장·번역한 직후, 연관 wiki 페이지를 즉시 확인하고 통합까지 완료해야 ingest 완료로 간주한다
  - "일단 raw만 저장"으로 끝내지 않는다 — raw 저장 = 중간 단계, wiki 반영 = 완료 단계
- **제목이 달라도 내용이 같으면 하나의 wiki 페이지로 통합** — 파일명·문서 제목이 기준이 아니라, 다루는 주제·개념이 기준이다
  - 예: `swaddleup-review.md`(한국어 리뷰)와 `【공식】스와들업...`(일본어 공식 페이지)는 제목이 달라도 [[스와들업]] 하나로 통합
  - 예: `bebeswan.md`(영어 제목)와 `베베스완-리뷰.md`(한국어 제목)는 [[베베스완]] 하나로 통합
  - 예: 회사소개서 PPT → 기존 [[마더스마일-회사개요]]에 통합
- 내용이 겹치는 raw 파일이 여럿이면, 각 파일에서 **새로 추가되는 정보만** 추출해 wiki에 반영 (중복 서술 금지)
- 관련 내용이 분산된 경우 [[위키링크]]로 연결 — 같은 내용을 여러 페이지에 반복하지 말고 한 곳에 두고 링크

**언어 무관 원칙**:
- raw 파일 언어(한국어·일본어·영어 등)는 wiki 페이지 배치에 영향을 주지 않는다 — **내용의 주제**만 본다
- **볼트 내 모든 텍스트는 한국어로** — wiki 페이지 본문, raw 파일 본문, YAML frontmatter(title·author·description 등) 전부 포함
- 고유명사·브랜드명은 한국어 표기 기준으로 작성: `스와들업`, `라쿠텐`, `머니포워드`
- 단, 원문 인용이 필요한 경우 `> 인용` 블록 안에서만 원어 사용 허용
- **YAML frontmatter 작성 주의**: `author` 필드에 `[[링크]]` 형식 절대 사용 금지 — Obsidian이 빈 페이지를 자동 생성함. 일반 텍스트로만 작성
  - ❌ `author: - "[[株式会社マザースマイル]]"`
  - ✅ `author: - "마더스마일 주식회사"`

**기계번역 클리핑 처리 주의사항**:
웹 클리핑 raw 파일은 Obsidian Web Clipper 등이 일본어를 자동 번역한 경우가 많다. 아래 오역 패턴을 반드시 확인하고 수정한 뒤 wiki에 반영한다.

| 오역 패턴 | 원문 | 올바른 번역 |
|----------|------|-----------|
| `기적의 호두` | くるみ (swaddle/wrap) | `기적의 속싸개` |
| `포옹 끈` / `포옹끈` | 抱っこ紐 | `아기띠` |
| `사랑 나무` | ラブツリー (Love Tree) | `러브트리` |
| `복귀` (육아 문맥) | はき戻し | `토출 방지` |
| `베이비 버스` | ベビーバス (bath) | `아기 목욕` |
| `고양이 포스` | ネコポス (배송 서비스) | `네코포스` |
| `스와돌업` | スワドルアップ | `스와들업` |
| `러브 트리` (띄어쓰기) | ラブツリー | `러브트리` |
| 가격 뒤 `년` | ¥1,980 (엔 단위) | `¥1,980` |

- 파일명에 오역이 포함된 경우 CLAUDE.md 명명 규칙에 따라 파일명도 수정한다
- 위 표에 없는 새 오역 패턴을 발견하면 이 표에 추가한다

**raw 파일 유형별 처리 방식**:

| raw 유형 | 판별 기준 | wiki 처리 방식 |
|----------|---------|--------------|
| 문서 (PDF·DOCX·XLSX) | 텍스트 전문 | 내용 그대로 구조화 |
| 기사·웹 클리핑 | URL 출처 포함 | 핵심 내용 + 출처 링크 |
| 동영상 (YouTube 등) | 영상 URL·제목 포함 | **핵심 요약** (타임스탬프 + 3~5개 bullet) + 접힌 스크립트 블록 |
| 양식 파일 (DOCX·XLSX) | 빈 기입란 포함 | wiki 페이지에 `## 다운로드` 섹션 추가 |

**파일 삭제 기준** (즉시 삭제, 사용자 확인 불필요):
- **내용 없는 파일**: 제목만 있고 실질적 내용이 없는 wiki 페이지
- **링크만 있는 파일**: 본문이 iframe·외부 링크·이미지 링크만으로 구성된 파일 (예: SharePoint iframe 클리핑)
- **무제 파일**: `무제`, `무제 1`, `Untitled`, `Untitled 1` 등 이름 그대로인 파일
- **Obsidian 자동 생성 유령 파일**: raw/wiki 폴더 밖에 내용 없이 생성된 `.md` 파일
- 삭제 전 해당 파일을 참조하는 wiki 링크도 함께 제거

### 2. 질문하기 (Query)

**사용자가 할 일**: 자유롭게 질문한다.
> "반품 처리 절차가 어떻게 돼?" / "재고 줄이는 방법 알려줘"

**Claude가 할 일**:
1. `wiki/index.md`에서 관련 페이지 파악
2. 해당 wiki 페이지들 읽기
3. 출처 wiki 페이지를 명시하며 답변
4. 답변 자체가 새 지식이라면 wiki 페이지로 저장 제안

**규칙**:
- 답변 마지막에 항상 참조한 wiki 페이지 목록 표시
- wiki에 없는 내용이면 "위키에 없음 — raw 자료를 추가해주세요" 안내

---

### 3. 건강검진 (Lint)

**사용자가 할 일**: 주기적으로 (월 1회 권장) 요청한다.
> "위키 건강검진 해줘"

**Claude가 할 일**:
1. 볼트 전체 순회 (wiki/ + raw/ + 루트)
2. 아래 항목 점검 후 보고서 작성:
   - **고아 페이지**: index.md에 없는 wiki 페이지
   - **깨진 링크**: [[링크]]가 걸려 있으나 파일이 없는 경우
   - **중복 내용**: 같은 주제가 여러 페이지에 분산된 경우
   - **오래된 정보**: log.md 기준 6개월 이상 미업데이트 페이지
   - **takeaway 누락**: callout 블록이 없는 wiki 페이지
   - **외국어 잔존**: 히라가나·가타카나·한자가 포함된 파일 (raw YAML 포함)
   - **삭제 대상 파일**: 내용 없음·링크만·무제·유령 파일
   - **미처리 raw**: ✅ 마커가 없는 raw 파일
3. 수정 권고 목록 제시 (사용자 승인 후 수정)

## 동영상 페이지 형식

동영상 raw 파일을 ingest할 때는 아래 구조를 사용한다:

```markdown
# 영상 제목

> [!tip] 서포트팀 이사 관점 핵심 takeaway
> ...

## 핵심 요약
- `00:00` 도입부 — 한 줄 요약
- `03:20` 핵심 포인트 1
- `07:45` 핵심 포인트 2
- `12:10` 결론

🔗 [영상 보기](YouTube URL)

> [!note]- 스크립트 보기
> 관련 구간 스크립트 또는 상세 발언 내용.
> 여러 줄 작성 가능.

## 관련 페이지
...
```

- `> [!note]-` 의 `-`가 핵심 — 기본 **접힌 상태**로 표시, 클릭하면 펼쳐짐
- 스크립트는 전체 녹취가 아닌 **핵심 구간만** 발췌
- 타임스탬프는 YouTube URL에 `?t=초` 형식으로 딥링크 가능: `[03:20](https://youtu.be/xxx?t=200)`

## 카테고리 정의

| 카테고리 | 설명 | 대상 raw 유형 |
|----------|------|--------------|
| 규정·정책 | 사내 규정, 업무 프로세스, 정책 문서 | 규정 문서 |
| 아이디어 | 사업 아이디어, 개선안, 메모 | 아이디어 메모 |
| 업계·시장 | 유아용품 업계 동향, 경쟁사, 트렌드 기사 | 링크·기사 |
| 매뉴얼·절차 | 업무 매뉴얼, 시스템 사용법, 체크리스트 | 매뉴얼 |
| 교육·학습 | 강연, 교육 자료, 책, 세미나 메모 | 교육 자료 |
| 프로젝트 | 진행 중·완료된 프로젝트 기록, 의사결정 이력 | 프로젝트 파일 |
